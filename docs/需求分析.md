# 需求分析

## 子问题分解

本项目围绕“多本教材压缩到不超过 30% 精华版本并保持教学效果”拆成六个子问题：

1. 教材解析：把 PDF、Markdown、TXT、DOCX 转成统一的教材-章节结构。
2. 知识点粒度定义：以章节内可独立讲授、可被问答引用的概念、机制、现象、方法作为知识点。
3. 重复判定：先按知识点名称规范化和字符 ngram 相似度找候选，再用 LLM 输出或系统规则给出 merge/keep/remove 决策。
4. 教学连贯性保障：保留唯一知识点，合并重复知识点时保留较完整定义，并保留图谱关系和 RAG 原文引用。
5. 压缩比计算：`整合后知识点定义总字数 / 原始教材正文总字数`，系统在整合统计中展示。
6. RAG 问答：所有回答都基于检索 chunk，并返回教材、章节、页码和原文片段。

## P0 功能边界

- 多格式加载：首版支持 PDF、Markdown、TXT、DOCX；Excel 不作为 P0 阻塞。
- PDF 解析：使用 PyMuPDF 逐页提取文本，按“第X章/Chapter X”识别章节；无法识别时按页段兜底。
- 知识图谱：优先调用配置的大模型服务抽取 JSON；失败时用关键词启发式兜底，保证演示闭环不中断。
- 跨教材整合：首版用 TF-IDF 字符 ngram 相似度分组，再生成解释性决策。
- RAG：首版用 TF-IDF + BM25 + 明确概念短语命中的混合检索，保证用户提到教材中存在的核心概念时优先召回含该概念的 chunk；后续可替换 embedding 模型。
- 教师反馈：首版支持自然语言覆盖匹配到的整合决策，并保留会话历史。

## 分块策略

RAG 按章节正文切成约 700 字 chunk，相邻 chunk 重叠 80 字。这个粒度兼顾三点：

- 单个 chunk 足够包含课程概念定义和上下文。
- top-5 注入 LLM 时不会过长。
- 80 字重叠降低定义跨 chunk 被切断的概率。
- 查询重排序会降低“概念、定义、是什么”等泛化问法词的干扰，并提高明确实体词在原文中的精确命中权重。

## RAG Benchmark

按赛题要求，项目需要自建 20-50 道 benchmark 题，覆盖事实、对比、推理/关联和跨教材来源定位，并为每题记录预期答案提示、教材、章节、页码和 chunk 作为 ground truth。评测指标包括 Recall@5、MRR、答案准确率、引用命中率、证据命中率、平均响应时间和估算 token 成本。

自动优化流程对比 chunk size 300/500/800/1200、重叠长度、稀疏检索模型和短语 rerank 开关，选择综合指标最优的配置写入 `src/backend/app/rag_defaults.json`。完整评测结果由 `scripts/rag_benchmark.py` 生成到 `docs/RAG Benchmark.md`，运行细节保存在被 Git 忽略的 `data/cache/rag_benchmark/latest.json`。

## 创新需求

除 P0/P1 基础功能外，项目把 F 维度创新拆成四类可验收需求：

1. 技术创新：RAG pipeline 必须有自建 benchmark 和自动调参结果，避免只凭经验选择分块策略。
2. 工程创新：LLM 抽取、RAG 回答和部署都必须有兜底路径，避免演示因外部服务或平台差异中断。
3. 体验创新：教师应能在同一工作台完成上传、看图谱、看整合、问 RAG、给反馈、生成报告。
4. 可解释创新：每个整合决策和 RAG 回答都必须保留来源教材、章节、页码，便于教学复核。

这些创新点的实现和证据集中记录在 `docs/创新点说明.md`，并在 `docs/Agent 架构说明.md` 中映射到架构设计取舍。

## 风险

- 大体量 PDF 完整解析和全章 LLM 抽取耗时较高。
- 首版图谱和整合算法偏工程可用，不等同于人工教研级别精度。
- 没有真实 embedding 模型时，跨语言同义词需要依赖 LLM 或别名字典补强。
